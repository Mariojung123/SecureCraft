[01] Why We Build

Motivation

Most security courses teach theory. Learners read about SQL injection but never inject anything. They memorize OWASP categories without ever writing a payload. The gap between knowing and doing is where developers ship vulnerable code.

PROBLEM_01
Theory Without Practice

Reading about XSS doesn't build the reflex to escape user input. Muscle memory requires repetition — real inputs, real exploits.

PROBLEM_02
No Safe Attack Environment

Learners can't freely practice offensive techniques. Setting up a local vulnerable app is complex; cloud VMs cost money.

PROBLEM_03
Generic Feedback

Existing tools tell you "this code is vulnerable" but rarely explain the exact line, how an attacker exploits it, or how to fix it.

PROBLEM_04
Zero AI Guidance

No interactive AI tutor that understands the specific code submitted, explains the vulnerability in context, and answers follow-up questions.

Learn by Breaking

The core loop: write code, see it attacked in a real container, read Claude's exact-line analysis, chat with the AI tutor about what went wrong. The experience is adversarial by design — the same code gets both analyzed for vulnerabilities and actively exploited in the sandbox simultaneously.

  • Adversarial learning — submit code → AI analyzes AND attack script runs in parallel
  • Live exploit — real HTTP request with attack payload, not a simulation
  • Contextual AI — Claude's report knows your exact submission and guides your fix
  • Air-gapped sandbox — Docker container with network=none; no internet escape possible

[05] System Design

Architecture

Four-layer system. Frontend never touches Docker or AI directly — all orchestration happens in the backend service layer with a strict separation between HTTP handling, business logic, sandbox management, and AI calls.

Frontend
ChallengeList.jsx ChallengeDetail.jsx Sandbox.jsx Report.jsx CodeEditor.jsx AiChat.jsx
↓ axios via api.js · polls /api/sessions/:id until done
API Routes
/api/challenges /api/submit /api/sessions/:id /api/report/:id /api/attack /api/chat
↓ validates request → dispatches to service layer
Services
submit_service.py ai_analyzer.py orchestrator.py
↓ concurrent threads: AI analysis ‖ Docker sandbox build+run
Runtime
Docker SDK attack.sh check.py Claude API
  • Submission pipeline — submit_service runs two concurrent threads: (1) build_image → run_validation via attack.sh, (2) run_sandbox for live iframe; results merged into session dict
  • AI + regex fallback — ai_analyzer calls Claude for vulnerability analysis; if JSON parse fails, falls back to regex-based vuln_patterns from meta.json
  • Session store — in-memory dict with threading.Lock; session lifecycle: processing → done / error → auto-cleanup after TTL
  • Problems directory — each challenge is self-contained: meta.json, skeleton code, solution, Dockerfiles, attack.sh, check.py

02

Monaco Editor — VS Code Quality in Browser

Skeleton code loads directly into a Monaco Editor instance (same engine as VS Code). Syntax highlighting, bracket matching, and line numbers out of the box — no installation, no friction. Learners write defensive code the same way they'd write production code.

Monaco Editor 4.7 Syntax Highlight Multi-language
03

Docker Sandbox — Real Exploit, Safe Environment

Submitted code runs in a hardened Docker container: network=none, read-only filesystem, 128 MB memory cap, all Linux capabilities dropped. The attack.sh script fires real HTTP payloads at the running container and returns VULNERABLE or SECURE. The live sandbox iframe lets learners manually test their own payloads too.

network=none 128MB limit no-new-privileges 60s TTL
04

Claude AI Vulnerability Analysis

Claude receives the submitted code and challenge context, then returns a structured JSON report: is_vulnerable flag, severity badge (HIGH/MEDIUM/LOW/NONE), the exact vulnerable line numbers, a plain-English explanation of the exploit, the attack payload used, and a specific fix hint. Falls back to regex pattern matching if JSON parsing fails.

Claude API Vuln Lines Severity Badge Fix Hint Regex Fallback
05

Contextual AI Chat

After the report loads, learners can ask Claude follow-up questions. The system prompt is built from the AI report — Claude knows the challenge, the submitted code, the vulnerability found, and the severity. Multi-turn conversation history injected each round. "Why is line 7 dangerous?", "Show me a safer version" — all answered in context.

Claude API Report Context Multi-turn

Functional components + hooks only. Monaco Editor for VS Code-quality code input. Axios for async polling against the session endpoint.
Backend
Flask 3.1 (Python 3.11)
Official Anthropic Python SDK. Native Docker SDK integration. threading for concurrent AI + sandbox execution on submit.
Sandbox
Docker + Docker SDK
Full container lifecycle management: image build, run, attack, auto-cleanup. Per-challenge Dockerfiles for Python, PHP, and Java. attack.sh scripts for automated exploit verification.
AI
Claude API (Sonnet 4.6)
Structured JSON vulnerability reports with exact line numbers. Context-aware tutoring chat built from the report. Regex fallback ensures zero silent failures.
Database
In-memory Sessions + MariaDB
Session data stored in a thread-safe Python dict for the current MVP. MariaDB provisioned via Docker Compose for future auth + progress tracking.
Testing
pytest — 42 tests
Unit tests for AI JSON parsing, regex fallback, route responses. Docker and Claude calls fully mocked — tests run without external dependencies.

[09] Security Hardening

Sandbox Constraints

Every user submission runs inside a hardened Docker container. The attack surface is zero by design — learners can write and exploit any vulnerability they want, but the container cannot reach the host system or the internet.

--network
none
Air-gapped — no internet, no host network access
--memory
128m
Caps DoS / memory bomb attacks
--cpu-quota
50000 (50%)
One core max; prevents CPU starvation
--read-only
true
No filesystem writes; rootkit can't persist
--security-opt
no-new-privileges
Blocks setuid / privilege escalation
--cap-drop
ALL
All Linux capabilities removed
--user
1000:1000
Non-root process inside container
auto-cleanup TTL
60 seconds
Container removed automatically after session expires
